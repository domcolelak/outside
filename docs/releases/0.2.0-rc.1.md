# OUTSIDE 0.2.0-rc.1 release candidate

This candidate includes a production-like staging topology and machine-generated launch validation workflow. These prove repository-controlled deployment and recovery behaviours in an isolated runner; they do not establish public ACME, managed PITR, real provider delivery, authorized-domain quality, regional placement, legal approval, independent penetration testing, or production SLO history.

## Release identity

The formal artifact workflow records these values from the exact checked-out commit:

- application version: `0.2.0-rc.1`;
- Terraform provider version: `0.2.0-rc.1`;
- database schema head: `20260717020000_billing_event_order`;
- Git commit and UTC build timestamp;
- verified CI run URL;
- application and migration container image IDs;
- SHA-256 digests for both deployable container archives, the Terraform provider archive, and the release manifest.

Do not substitute a locally rebuilt image for the archived artifact without creating a new manifest and repeating validation.

## Environment changes

Required production values remain:

- `OUTSIDE_STORAGE_MODE=database`;
- TLS PostgreSQL `DATABASE_URL`;
- canonical HTTPS `APP_URL`;
- independent `AUTH_SECRET`, `OUTSIDE_VERIFY_SECRET`, `CRON_SECRET`, `RESEND_API_KEY`, and `EMAIL_FROM`.

Feature-specific values:

- `GUARDIAN_ENCRYPTION_KEY` for Guardian delivery credentials;
- `ENTERPRISE_ENCRYPTION_KEY`, `ENTERPRISE_PROVISIONING_TOKEN`, and `AUDIT_IP_SALT` for Enterprise;
- complete Google, Stripe, and provider credential sets only when those features are enabled;
- OTLP endpoint, headers, service name, and export interval for operational metrics.

The image supplies `OUTSIDE_APP_VERSION`, `OUTSIDE_GIT_SHA`, and `OUTSIDE_BUILD_TIME`. Operators must not override them unless they are deliberately repackaging and re-validating the artifact.

## Migration notes

There are 12 ordered Prisma migrations from the baseline through `20260717020000_billing_event_order`. Run the matching migration image once:

```bash
docker load < outside-migrator-0.2.0-rc.1.docker.tar.gz
docker run --rm --env DATABASE_URL outside-migrator:0.2.0-rc.1 migrate deploy
```

Before applying to customer data:

1. record a managed restore point;
2. restore representative data into isolated staging;
3. apply the migration image;
4. verify `_prisma_migrations`, tenant counts, evidence history, audit chains, billing state, and application readiness;
5. retain timings and approver identity in the release record.

Never use `prisma db push` in staging or production.

## Deployment

Load the application archive, start it as a non-root read-only container with a writable `/tmp`, inject secrets at runtime, and terminate HTTPS at a trusted proxy that replaces forwarding headers. Apply migrations before starting new instances. Verify:

1. `/api/livez` returns version, commit, and build timestamp matching the manifest;
2. `/api/readyz` is ready with database persistence;
3. production security headers are present over HTTPS;
4. one deterministic demo scan completes without persisting synthetic data;
5. authenticated schedulers run and report non-degraded outcomes;
6. OTLP metrics and structured logs arrive in the staging observability backend.

## Rollback

- Stop traffic to the new image and restore the prior immutable image.
- Do not automatically reverse schema changes. Prefer a reviewed roll-forward correction.
- If a failed migration may have changed data, stop workers and egress, inspect migration state, and restore the pre-release snapshot when required.
- Preserve queue rows and let leases expire; verify idempotency before resuming notifications or integrations.
- Confirm liveness, readiness, scheduler ownership, billing webhooks, and queue age after rollback.

## Promotion blockers

This candidate must not be promoted to a stable public-production release until:

- the exact artifact is deployed to a persistent HTTPS staging environment;
- critical customer and operational workflows are exercised there;
- managed backup/PITR, key-recovery, failure, and update/rollback drills are recorded;
- representative performance measurements establish provisional pilot SLOs;
- external security and legal statuses are explicitly recorded.
